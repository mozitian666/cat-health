## 幻灯片 3：项目背景 - 研发意义
- **健康焦虑与管理痛点**：
  - 现代人生活节奏快，亚健康、肥胖问题日益严重。
  - 传统饮食记录应用（如 keep, Apple health）操作繁琐，用户难以坚持。
  - "记录"本身是反人性的，枯燥的数据录入导致用户流失率极高。
- **游戏化 (Gamification) 的潜力**：
  - 将枯燥的健康管理转化为有趣的“养成游戏”。
  - 通过“电子宠物”的情感羁绊，激励用户持续打卡，将被动记录转变为主动投喂。

---

## 幻灯片 4：项目背景 - 国内外现状
- **国外现状**：
  - **Noom**：主打心理学干预，但缺乏趣味性互动。
  - **Habitica**：RPG 风格的任务管理，但未专注于饮食 AI 识别领域。
- **国内现状**：
  - **薄荷健康/Keep**：功能强大但工具属性过重，缺乏情感化陪伴。
  - **电子宠物类 APP**：多为纯娱乐，缺乏与现实生活数据的深度挂钩。
- **存在的问题**：
  - **数据录入成本高**：手动搜索食物、估算重量劝退用户。
  - **反馈延迟**：吃了健康的食物，身体变化需要很久才能看到，缺乏即时满足感。
  - **同质化严重**：市面产品多为“数据仪表盘”，缺乏温度。

---

## 幻灯片 5：拟用方法 - 核心理念
- **AI 赋能效率**：
  - 利用计算机视觉 (CV) 技术，实现“拍照即记录”，将录入时间从 3 分钟缩短至 10 秒。
- **情感化反馈机制**：
  - **映射机制**：用户的每一次健康行为（吃蔬菜、喝水）直接转化为电子小猫的成长数值（经验值、毛发光泽）。
  - **负反馈调节**：暴饮暴食会导致小猫变胖、生病，利用用户的同理心约束不健康行为。
- **闭环系统**：
  - 记录 (Input) -> AI 分析 (Process) -> 小猫成长/健康建议 (Output) -> 持续激励 (Feedback)。

## 幻灯片 6：技术路线 - 架构设计
- **前端 (Client)**：
  - **框架**：React + Vite (保证高性能与快速迭代)。
  - **UI/UX**：TailwindCSS (现代化响应式设计) + Lucide Icons。
  - **跨平台**：支持 Web 端及通过 Electron 封装的桌面客户端。
- **后端 (Server)**：
  - **运行环境**：Node.js + Express (轻量级、高并发)。
  - **数据库**：SQLite (开发期) / MySQL (生产期)，使用 Sequelize ORM 进行数据建模。
  - **文件服务**：Nginx 反向代理，处理静态资源与图片上传。
- **部署运维**：
  - **容器化**：Docker + PM2 进程守护，确保服务 7x24 小时高可用。
  - **服务器**：Linux (Ubuntu)，配置 SSL 安全证书。

---

## 幻灯片 7：技术路线 - 核心功能实现
1. **AI 饮食识别模块**：
   - 图像上传 -> 预处理 -> 调用 AI 识别接口 (Google Vision/百度 AI) -> 返回营养数据。
2. **电子宠物状态机**：
   - 数值模型：`Energy` (活力), `Weight` (体重),, `Exp` (经验)。
   - 触发逻辑：
     - `Drink Water` -> `Energy +10`
     - `High Protein Diet` -> `EXP +2`
     - `High Calorie` -> `Weight +0.1`
3. **可视化数据看板**：
   - 使用 Chart.js 绘制营养环形图，实时展示蛋白质/碳水/脂肪比例。

## 幻灯片 8：可行性分析
- **技术可行性**：
  - 前后端分离架构成熟，社区生态丰富。
  - AI 图像识别 API 已非常普及，识别准确率高，接入成本低。
  - 团队已完成从数据库设计到 Docker 部署的全流程验证。
- **经济可行性**：
  - 初期采用轻量级服务器和开源技术栈，研发成本极低。
  - 商业模式清晰：基础功能免费，高级皮肤/道具/更精准的 AI 分析可作为增值服务。
- **操作可行性**：
  - 界面简洁（Home/Scan/Cat 三段式导航），符合用户直觉。
  - 拍照上传大大降低了用户的使用门槛。
