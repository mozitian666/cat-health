痛点分析：传统选课的困境

  • 资源冲突： 教室与时间安排极易发生重叠，人工排课效率低。
  • 规则复杂： 学分上限控制、先修课程检查难以通过人工实时验证。
  • 信息孤岛： 学生无法实时获取课程剩余容量，导致选课失败率高。

核心业务流：选课过程

  1. 1. 提交请求： 学生发起选课申请。
  2. 2. 资格检查： 系统检查是否已修先修课。
  3. 3. 冲突检测： 检查时间表是否有重叠。
  4. 4. 容量锁定： 检查剩余名额并锁定。
  5. 5. 事务提交： 更新数据库，生成选课记录。
演讲备注：这是系统的核心业务流——选课。从提交请求到最终事务提交，系统按顺序执行资格、冲突、容量三道关卡的原子性检查，任何一环失败都会回滚，保证数据一致性。

多对多关系的实现

学生与课程之间是典型的多对多关系。

  • 挑战： 直接关联无法记录"选课时间"、"成绩"等额外信息。
  • 方案： 引入 Enrollment 作为关联实体（Join Entity）。
  • 优势： 解耦了学生与课程，便于扩展更多选课属性。

Student

ID, Name

Enrollment

Student_ID, Course_ID, Date

Course

ID, Title

演讲备注：为了优雅地处理学生与课程的多对多关系，我们引入了Enrollment实体。这不仅解决了关联问题，还为未来扩展成绩录入、选课状态等属性预留了空间。

先修课程设计：自关联

课程之间的依赖关系通过自关联（Self-Reference）实现。

  • 模型： Course 表中包含 prerequisites 字段。
  • 关系： Many-to-Many 自关联。
  • 逻辑： 选课前递归检查 prerequisites 中的课程是否已在 completed_courses 中。
[图示：Course A (Advanced) ---> Depends on ---> Course B (Basic)]
演讲备注：系统的一个难点是先修课逻辑。我们利用Course表的自关联设计，成功建模了复杂的课程依赖树，支持多级先修关系的自动校验。

数据库 Schema 设计

Table: app_user
- id (PK), username, password, role

Table: course
- id (PK), title, credits, capacity, schedule, teacher_id (FK)

Table: course_prerequisites
- course_id (FK), prerequisite_id (FK)

Table: enrollment
- id (PK), user_id (FK), course_id (FK), created_at

采用规范化设计（3NF），减少数据冗余。

演讲备注：这是我们的数据库Schema设计。通过四张核心表和严格的外键约束，我们实现了第三范式（3NF），保证了数据的完整性和低冗余。

亮点功能 1：21 学分智能上限

  • 规则： 每个学生每学期选课总学分不得超过 21 分。
  • 实现：
    • Service 层在每次选课前计算 currentCredits + newCourseCredits。
    • 若 > 21，抛出异常并阻断事务。
[截图：前端弹出警告 'Cannot enroll: Credits exceed limit of 21']
演讲备注：核心功能展示。首先是"21学分智能上限"。系统在Service层强制拦截超额选课，这不仅是简单的数字限制，更是对教学质量的系统级保障。
